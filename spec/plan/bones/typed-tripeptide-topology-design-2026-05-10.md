This is the architectural design for a typed-substrate layer on the
DFT-tripeptide side that mirrors the protein-side `LegacyAmberTopology`
+ `AtomSemanticTable` machinery. The goal is to make atom matching
between DFT records and protein residues a typed-identity intersection
rather than the current heuristic element-pattern + nearest-distance
matching.

**Status:** designed, not implemented. Estimated 5–6 hours of focused
work (two sessions). The current calculators (`TripeptideBackboneShieldingResult`,
`TripeptideNeighborShieldingResult`) ship via heuristic matching on
sidechain atoms beyond Cβ; this design replaces that with typed
matching at both ends.

**Authorship:** Plan agent design (2026-05-10) + user enrichment for
per-atom pose metadata (governing dihedral + rotation axis + bond
parent). Both are folded into a single durable spec here.

## Executive summary

We have one fully typed substrate (protein side via `LegacyAmberTopology`
+ `AtomSemanticTable`) and one untyped substrate (DFT side via
`TripeptideDftRecord::atoms`). The current pose assembler bridges them
with heuristic matching, which has silently swapped sidechain atoms
(SER OG ↔ backbone O before backbone role-pinning was added).

The architectural fix is to stand up a typed substrate on the DFT side
that uses **exactly the same `AtomSemanticTable` records and lookup
machinery as the protein side**, and rewrite matching to be
typed-identity ↔ typed-identity. No adapter, no bridge — the protein
side's typed object IS the matching vocabulary, and the DFT side just
needs to populate the same struct per row.

Each per-DFT-row-position entry carries not just `AtomMechanicalIdentity`
but also pose metadata (governing dihedral, rotation axis, bond
parent) so the per-atom residual can be decomposed into a chi-rotation
component (recoverable systematically) and a geometry component (real
PM6-vs-protein bond/angle residual) for the ML model.

## Architectural decisions

### Decision 1: Where does the typed tripeptide topology live?

**Recommended:** a free function `ComposeTripeptideSemantics(record)`
returning a `vector<AtomSemanticTable>` parallel to `record.atoms`,
plus storage on the record itself (`TripeptideDftRecord::atom_semantics`)
filled lazily on first need.

**Rejected:** a new `TripeptideTopology` class mirroring
`LegacyAmberTopology` in shape. `LegacyAmberTopology` holds bonds +
rings + per-atom semantics + FF invariants for a *protein* with
conformations, ConformationResults attached, etc. A tripeptide DFT row
is a 3-residue capped peptide with none of that. The shape mismatch
makes the class an adapter; per the "no adapter classes" rule in
PATTERNS.md it would be wrong.

The protein side does it the same way: `LegacyAmberTopology` holds
`vector<AtomSemanticTable> atom_semantic_`, populated by free function
`ComposeAtomSemantic(...)` at construction. We adopt the same shape.

### Decision 2: Build-time table vs runtime perception?

**Recommended:** build-time positional table per `(central_residue,
frame_type)`, generated by extending `tools/topology/build_semantic_tables`
to also emit `src/generated/TripeptideOrderingTables.cpp`.

The user has verified ALA, SER orderings via psql. The orderings are
canonical and stable per `(residue, frame_type)`. Build-time positional
tables are the same shape as the existing `kAlaAtoms`, `kSerAtoms`,
etc. residue tables — just keyed on DFT-row position instead of
CCD-residue position.

**Rejected:** runtime bond perception via OpenBabel on (positions,
elements). It would:

- Add a second perception path (OpenBabel) alongside the protein
  side's RDKit perception. Per `feedback_canalisation_is_expertise`
  and the substrate-side string barrier, the runtime library should
  not perceive chemistry — perception happens at build time, the
  runtime consumes typed tables.
- Pay per-row perception cost on every query. DB has ~6,000 rows for
  SER, ~100K for most residues — perceiving them at runtime is wasted
  work; they're already perceived at deposit time.
- OpenBabel is technically linked into the runtime but is used at
  PDB-load time, not as a substrate-composition primitive. Adding
  it to substrate composition would erode the discipline.

**Hybrid (build-time primary, runtime perception as cross-check)** —
rejected. Adds machinery that earns its keep only if we suspect the
build-time table will be wrong. We don't. A test (Part 7 below) is the
right place for cross-check.

### Decision 3: Frame_type discriminator

The build-time table must be keyed on `(central_residue, frame_type)`
because the SER PBE re-run (`orca_input_orientation`) may have a
different atom ordering than the Gaussian standard orientation OPBE
rows for the same SER residue.

**Open question, must verify before committing:** are ORCA SER rows
in the same atom order as Gaussian SER rows? If yes, one SER ordering
table. If no, two — small additional cost.

**Action:** Phase 1 step 1, psql one row of each combination per
residue × frame_type and emit elements.

### Decision 4: Per-position pose metadata (user enrichment)

Each ordering-table entry is **not just** `AtomMechanicalIdentity` —
it's a wider record:

```cpp
struct TripeptidePositionEntry {
    AtomMechanicalIdentity identity;
    GoverningDihedral      governing;       // None | Phi | Psi | Chi1..4
    int                    rotation_axis_a; // atom index within residue
    int                    rotation_axis_b; // atom index within residue
    int                    bond_parent;     // heavy-atom parent index in residue
    // (bond_length and bond_angle for position prediction — optional, defer)
};
```

New `GoverningDihedral` enum lives in `SemanticEnums.h` (sibling of
`BackboneRole`, `Locant`, etc.):

```cpp
enum class GoverningDihedral : std::uint8_t {
    None  = 0,   // Position fixed by backbone geometry (BB N/CA/C/O, HA).
    Phi   = 1,   // Position depends on backbone φ.
    Psi   = 2,   // Position depends on backbone ψ.
    Chi1  = 3,   // Position depends on χ₁ rotation about CA-CB.
    Chi2  = 4,   // χ₂ rotation about CB-CG (residue-specific).
    Chi3  = 5,   // χ₃ rotation about CG-CD.
    Chi4  = 6,   // χ₄ rotation about CD-CE (LYS, ARG).
};
```

Per-residue, the governing dihedral is determined by walking from CA
outward along the heavy-atom chain. Atoms past the first χ bond are
governed by χ₁; past the second χ bond by χ₂; etc. Backbone N/CA/C/O
and HA are `None` (pure geometry). Cβ is `Chi1` (it pivots around
CA-CB as χ₁ rotates — actually wait, Cβ's position is fixed once the
backbone is fixed; χ₁ rotates atoms BEYOND CB. Need to be careful here
— for SER, Cβ itself is `Chi1` because the CA-CB bond rotates around
the *axis defined by CA-CB-OG*, and Cβ's position is one endpoint of
that axis. The HB hydrogens on Cβ are also `Chi1`. The OG and HG are
`Chi1` (one bond past CA-CB). For LEU, Cβ is the CA-CB pivot,
HB2/HB3 are `Chi1`, CG is `Chi1`, CD1/CD2 are `Chi2`, etc.). This
needs careful walk-through per-residue when authoring the TOML.

`rotation_axis_a`, `rotation_axis_b` are indices into the **protein
residue's** atom list (so the consumer can look up the bond axis in
protein space). For Cβ in SER: `(rotation_axis_a=res.CA,
rotation_axis_b=res.CB)`.

`bond_parent`: the heavy-atom parent. For HB2 of SER: `bond_parent =
res.CB`. For HG: `bond_parent = res.OG`. Used for atom-position
prediction and for residual decomposition.

### What this metadata buys

1. **Per-atom residual decomposition.** The current `residual_vec` is
   a single Vec3. With governing dihedral metadata, the consumer can
   decompose:

   - **Chi-rotation component**: projection onto the rotation axis
     direction. Recoverable systematically — corresponds to |actual χ
     − grid χ| × radius. The model can correct for this with a
     learned per-residue χ-grid offset.
   - **Geometry component**: orthogonal — real PM6-vs-protein bond
     length / angle differences. The model attends to this as a
     non-recoverable residual.

   Two cleaner ML features than one mixed Vec3.

2. **Search-time pre-scoring.** On chi-fallback queries (residue's
   actual χ isn't on the 20° grid), instead of blindly dropping the
   deepest chi axis, score candidate DB rows by predicted per-atom
   angular residual:

   `score = Σ_atoms_in_residue (weight_atom × |actual χ_gov − grid χ_gov|)`

   Pick the best candidate geometrically rather than dropping χ_4
   first. Reduces residual on K/R atoms substantially (they have
   χ_3/χ_4; a small residual on χ_3 plus dropping χ_4 may be worse
   than keeping χ_4 and accepting more χ_3 residual).

3. **Inverse-search use case.** "Find the DB row that minimises
   residual at THIS atom" becomes an O(N_rows) scan rather than a
   grid walk. Useful when we want to cross-validate against the
   separate Larsen H-bond grid (which has its own `(rOH, θ, ρ)` scan)
   — for an atom-of-interest A, pick the DB row whose A position
   matches the protein's A position best, regardless of which (φ, ψ,
   χ) grid point it hits.

**Architectural cost:** small. ~5 ints per entry × ~30 atoms per
residue × 19 residues × 2 frame types = ~22 KB compile-time data.

### Decision 5: Persistence vs lazy

**Recommended:** lazy per-record. The semantic vector is small (~5 KB
per row); composition is cheap once the build-time table is loaded.
For a 76-residue protein × ~228 DFT queries (Central + N-cap + C-cap
per residue) × ~60 semantics records ≈ ~14K AtomSemanticTable records
total per protein — milliseconds.

`QueryNearest` returns the record with `atom_semantics` empty; the
pose assembler (or the calculator) fills it on first use:

```cpp
if (rec.atom_semantics.empty()) {
    rec.atom_semantics = ComposeTripeptideSemantics(rec);
}
```

### Decision 6: New matching API

```cpp
struct TripeptideAtomMatch {
    int                    dft_atom_idx     = -1;
    std::size_t            protein_atom_idx = SIZE_MAX;
    AtomMechanicalIdentity identity;     // same on both sides
    GoverningDihedral      governing;    // copied from the DFT-side entry
    bool                   ok = false;
};

std::vector<TripeptideAtomMatch> MatchTypedTripeptide(
    const Protein&                            protein,
    std::size_t                               protein_residue_idx,
    const TripeptideDftRecord&                rec,
    const std::vector<AtomSemanticTable>&     dft_semantics,
    TripeptidePoseSide                        side);
```

`AssembleTripeptide` orchestrates:
1. `ComposeTripeptideSemantics(rec)` if empty.
2. `MatchTypedTripeptide(...)` → per-atom typed pairs.
3. Kabsch on the BB N/CA/C triple from the typed matches.
4. For each match, transform DFT position + tensor → emit
   `AlignedDftAtom` with residual_vec (and optionally a
   `chi_residual_vec` and `geometry_residual_vec` if the consumer
   wants the decomposition).

The current `SubstrateRoleMatches` cross-check (each cap slot's
expected role) becomes redundant because both sides carry typed
identity. Keep it simplified to a single identity equality check as
defence-in-depth.

### Decision 7: ALA caps

ALA cap atoms ARE ALA atoms — the typed annotation is exactly `kAlaAtoms`
(10 entries: N, H, CA, HA, CB, C, HB1, HB2, HB3, O). The only thing
that differs between caps and central is the DFT-row position range.
The ordering table emits one entry per position; both N-term and
C-term caps reuse the same target identities.

**No `TerminalState::Nterm*`/`Cterm*` overlay** is needed for ALA caps
in tripeptides — the ACE/NME caps absorb the chain-end chemistry, so
the ALA caps' N, C, O backbone atoms carry their internal-chain
peptide-amide chemistry.

### Decision 8: ACE/NME caps

ACE = CH₃-C(=O)- methyl-acetyl cap. 6 atoms: carbonyl-C, carbonyl-O,
methyl-C, 3×methyl-H. NME = -NH-CH₃ methyl-amide cap. 6 atoms: N,
amide-H, methyl-C, 3×methyl-H.

These atoms **do not currently appear** in `LegacyAmberSemanticTables.cpp`.
The standard 20 residue tables cover internal residues; the four
cap-delta tables cover protein N-/C-terminal chain ends, NOT
acetyl/methylamide caps.

**Action:** add `kAceAtoms` (6 entries) and `kNmeAtoms` (6 entries)
to the generated semantic tables. ACE's carbonyl-C is
`BackboneRole::CarbonylCarbon` (it forms the peptide amide to ALA's
N); ACE's methyl is `Locant::None` with `PseudoatomKind::M`. NME's N
is `BackboneRole::Nitrogen` (peptide amide acceptor).

ACE/NME atoms are not matched against protein atoms (Larsen reads
tensors only at central + ALA caps). They exist purely to anchor the
ordering and feed any future bond-graph cross-check.

**Open question:** should ACE/NME live in the runtime `TerminalState`
enum? **Recommend defer.** The existing `TerminalState` covers
protein-chain caps; ACE/NME caps in tripeptides are a different
chemistry (no titration, no chain-end). Defer until a real PDB use
case wants them.

### Decision 10: Test strategy

Five test layers, in order:

**(a) Substrate parity test** — `tests/test_tripeptide_substrate_parity.cpp`.
Per `(residue, frame_type)` available in the DB: query one row, call
`ComposeTripeptideSemantics`, verify element pattern matches the
ordering table, verify identities populated for the central residue
equal the set returned by walking `kXxxAtoms`. **Canary** for ordering
table being wrong for any residue.

**(b) Build-time coverage** — extend `tests/test_topology_substrate.cpp`
(or sibling): for each AminoAcid × frame_type that has DB rows, assert
the ordering table has an entry; for each entry, assert `LookupBy`
returns non-null.

**(c) Heuristic-vs-typed agreement** — `tests/test_tripeptide_pose_assembler_typed.cpp`.
Load 1UBQ, run both heuristic and typed paths side-by-side. Assert:
- BB N/CA/C/O assignments agree on both paths per residue.
- Cβ assignments agree.
- Sidechain disagreements are LOGGED with full identity context.

**(d) SER sidechain regression** — `tests/test_tripeptide_ser_sidechain.cpp`.
Construct a SER DFT-record fixture where the heuristic is known to
mismap (the OG↔O swap case). Assert the typed path puts the SER OG
tensor on the protein's SER OG atom, not the protein's backbone O.

**(e) End-to-end** — re-run existing
`test_tripeptide_backbone_shielding.cpp` and
`test_tripeptide_neighbor_shielding.cpp`. Both must pass; matched-residue
count must be ≥ previous.

### Decision 12: Cost and risk

**Sizing:** 5–6 hours, two sessions.

Riskiest pieces:

1. **The build-time ordering tables.** Encoding the canonical Gaussian
   order per residue × frame_type. ALA, SER verified; other 17 need
   spot-check per residue. Discipline: psql dump → TOML → generator.
2. **Per-residue GoverningDihedral walk-through.** Need to carefully
   walk each residue's atom list from CA outward and assign
   governing dihedral. The chi_angle definitions in `AminoAcidType`
   are the basis; one-off thinking required for each residue.
3. **ACE/NME atom typed identities.** Hand-coded (not in CCD). Verify
   the typed-enum literals match the chemistry.
4. **ORCA SER ordering question.** Must verify before committing.
5. **Methyl-H pseudoatom collapse.** Three DFT atoms → one identity
   → three protein atoms case in `MatchTypedTripeptide`.

**Recommended hand-off shape:**

- **Session 1 (~2 hours):** substrate-side scaffolding. psql dump → TOML →
  generator extension → ACE/NME tables → substrate parity test green.
- **Session 2 (~3 hours):** runtime composer + assembler rewrite →
  parity tests → SER regression → cleanup → all existing
  tests green.

## Implementation order (step-by-step)

### Phase 1 — substrate scaffolding

1. **psql dump.** `scripts/dump_tripeptide_orderings.py` queries one
   row per `(central_residue, frame_type)` and emits per-atom (idx,
   element, atom_name) to JSON.
2. **Hand-author TOML.** `data/topology/tripeptide_orderings.toml`
   per `(residue, frame_type)` lists atom names in DFT-row order
   (atom names are the standard PDB names from `AminoAcidType::atoms`,
   so known typed inputs, not free-form strings).
3. **Per-residue governing-dihedral walk-through.** For each residue,
   walk atoms from CA outward and assign `GoverningDihedral` per atom
   based on which χ bond it's past. Document the walk in
   `data/topology/governing_dihedrals.toml` or inline in the same
   TOML.
4. **Extend generator** (`tools/topology/build_semantic_tables.cpp`):
   - Read tripeptide_orderings.toml + governing_dihedrals.toml.
   - For each `(residue, frame_type)`, walk the atom-name list, run
     `ParseAtomName` to get each `AtomMechanicalIdentity`, attach
     governing dihedral + rotation axis + bond parent, emit a
     `constexpr std::array<TripeptidePositionEntry, N>`.
   - Emit `kAlaCapOrdering`, `kAceCapOrdering`, `kNmeCapOrdering`.
   - Emit `kAceAtoms`, `kNmeAtoms` into the main residue table list.
5. **Regenerate.** `cmake -B build-gen -DNMR_BUILD_TABLE_GENERATOR=ON
   && cmake --build build-gen --target build_semantic_tables &&
   ./build-gen/tools/topology/build_semantic_tables ...`. Commit
   the regenerated `.cpp` files.
6. **CMake.** Append `src/generated/TripeptideOrderingTables.cpp` to
   `nmr_shielding` sources.
7. **Substrate parity test** — `tests/test_tripeptide_substrate_parity.cpp`.
   One query per residue × frame_type, assert element pattern matches.

### Phase 2 — runtime composer + typed matching

8. **Write `src/TripeptideTopology.{h,cpp}`.** Implement
   `ComputeTripeptideLayout`, `ComposeTripeptideSemantics`,
   `MatchTypedTripeptide`. Add to `CMakeLists.txt`.
9. **Rewrite `AssembleTripeptide`** in `TripeptidePoseAssembler.cpp`:
   - Call `ComposeTripeptideSemantics(rec)` if `rec.atom_semantics.empty()`.
   - Call `MatchTypedTripeptide` to get the typed pairing list.
   - Build Kabsch input from typed BB matches.
   - Per typed match, emit `AlignedDftAtom`.
   - Optionally: emit `chi_residual_vec` and `geometry_residual_vec`
     decomposition from `governing` field.
   - Delete the old `IdentifyCentralBackbone` / `IdentifyAlaCap`
     branches.
10. **Update consumers.** Both calculators are unchanged at the
    Assemble API level; verify they still compile.
11. **Parity test** — `tests/test_tripeptide_pose_assembler_typed.cpp`.
    1UBQ end-to-end, both paths, BB asserted identical, sidechain
    disagreements logged.
12. **SER sidechain regression** — `tests/test_tripeptide_ser_sidechain.cpp`.
13. **End-to-end re-run** — existing
    `test_tripeptide_backbone_shielding.cpp` and
    `test_tripeptide_neighbor_shielding.cpp` must pass.

### Phase 3 — cleanup

14. Delete dead exports from `TripeptidePoseAssembler.h` (move to
    anon namespace in `.cpp` if still needed internally).
15. Simplify `SubstrateRoleMatches` to single identity equality.
16. Update OBJECT_MODEL.md "Per-Result Minimum Output" with the new
    typed-DFT-substrate discipline.
17. Update PATTERNS.md with the cross-substrate typed-matching
    addition (if it earns a top-level discipline mention).

## Open questions to resolve before coding

1. **ORCA SER ordering** — verify Phase 1 step 1.
2. **ACE/NME in `TerminalState` enum** — recommend defer; surface to
   user before deciding.
3. **Methyl-H pseudoatom collapse semantics in `MatchTypedTripeptide`**
   — three DFT atoms → one identity. Spell out the per-atom-tensor
   handling (each H gets its own per-DFT-atom tensor; the identity
   collapse is just for matching, not tensor averaging).
4. **`Unknown` residue handling** — confirmed safe; the upstream

5. **Per-DFT-atom Hybridisation / BondOrderMask** — currently not
   stored in `AtomSemanticTable`; not needed by the tripeptide
   layer either. Pass-through.

## How to apply

When opening a session to land this:

1. Read this document end-to-end.
2. Phase 1 first. Don't skip the substrate parity test before Phase 2.
3. The current heuristic-based assembler at
   `src/TripeptidePoseAssembler.cpp` is the cheat sheet — don't
   rebuild from scratch. The math (Kabsch, tensor rotation,
   sidechain re-rotation) is correct. What changes is the matching:
   typed identity lookups replace element-walks and
   nearest-distance searches.
4. The build-time generator at `tools/topology/build_semantic_tables.cpp`
   is the work surface for Phase 1. Read its README first.
5. The OpenAI-5.5-strong architecture layout doc at
   `spec/plan/openai-5.5-strong-architecture-layout.md` is the
   broader substrate-discipline framing — re-read if anything about
   typed substrate is unclear.
