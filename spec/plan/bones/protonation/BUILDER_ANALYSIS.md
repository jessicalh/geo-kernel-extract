# Builder Gap Analysis

## 1. The Object Model As Built

A Protein in this system is the molecule's identity, divorced from geometry. It owns atoms via `vector<unique_ptr<Atom>>`, residues as value types in a `vector<Residue>`, rings via `vector<unique_ptr<Ring>>`, and bonds as value types. It also owns a `ProteinBuildContext` and a vector of `unique_ptr<ProteinConformation>`. The Protein is default-constructible, non-copyable, and non-movable. The non-movable constraint exists because every ProteinConformation holds a raw `const Protein*` back-pointer set at construction. Since Protein lives on the heap via `unique_ptr<Protein>` and never moves, these pointers remain valid for the Protein's lifetime.

A ProteinConformation is one geometric snapshot. Its constructor receives a `const Protein*` and a `vector<Vec3>` of positions. Internally it builds a `vector<ConformationAtom>` parallel to the Protein's atom list, with each ConformationAtom's position set const at construction and never changed. This parallelism is the central coupling in the system: every index into the Protein's atom list is simultaneously an index into the conformation's atom list, into the positions vector, into the spatial neighbour lists, into every per-atom accumulation array on every ConformationResult. A different atom count means a different Protein, not a different conformation.

Protein owns hydrogens because the atom list defines the molecule. Hydrogen atoms determine bond connectivity (parent_atom_index is set during DetectCovalentBonds), ring type (HIS tautomer is detected from which nitrogen carries hydrogen), and charge distribution (ff14SB assigns different charges to protonated vs deprotonated forms). The calculators traverse atoms by index, and every calculator that sums over atom pairs (Coulomb, McConnell, Dispersion) includes hydrogen atoms in its sums. Removing or adding a hydrogen changes every index downstream of its position in the atom vector.

Positions being const after construction means no result is ever stale with respect to geometry. The SpatialIndexResult builds KD-trees from the positions at construction; the GeometryResult computes ring centers, bond midpoints, and bounding boxes. Since positions never change, these derived quantities never need invalidation. This is a deliberate trade: new geometry costs a new conformation, but stale-data bugs are impossible.

Protein being non-movable is more conservative than strictly necessary. The PATTERNS.md records an earlier bug where move semantics left dangling back-pointers, which was solved by deleting move operations entirely rather than writing a back-pointer-fixing move constructor. The current code does not need move because both loaders create the Protein via `make_unique<Protein>()`, populate it in place, and return the `unique_ptr` without moving the Protein itself. The builder would follow the same pattern.


## 2. The Protonation Lifecycle

There are two distinct paths today, and neither offers protonation as a requestable step.

**Path 1: PDB loading (LoadProtein in PdbFileReader.cpp).** The function reads atoms from a PDB file via cif++, creates a default-constructed Protein, adds atoms and residues by iterating the cif::mm::structure polymers, calls `FinalizeConstruction(positions)` which runs CacheResidueBackboneIndices, DetectAromaticRings, and DetectCovalentBonds in that order, sets a minimal ProteinBuildContext, and creates a CrystalConformation. No protonation step occurs. If the PDB has hydrogens, they become atoms. If it does not, there are no hydrogen atoms and the molecule is incomplete for calculator purposes. For HIS ring detection, the code falls back to checking whether "HD1" and "HE2" atom name strings are present, which only works if the PDB was already protonated by some external tool.

**Path 2: ORCA run loading (LoadOrcaRun in OrcaRunLoader.cpp).** This reads an AMBER prmtop for the atom list and an XYZ file for positions. The prmtop was produced externally by tleap with ff14SB, which means protonation already happened outside our code. The loader reads AMBER residue labels (HID, HIE, HIP, ASH, GLH, etc.) and matches them to variant indices via the NamingRegistry's ToCanonical plus direct name comparison against the AminoAcidType variant table. It sets `protonation_variant_index` on each Residue during loading. Then it calls FinalizeConstruction, which uses these variant indices to determine ring types for HIS residues. The result is a complete Protein with hydrogens, correct ring types, and variant annotations, but the protonation decisions were made by tleap, not by our system.

**ProtonationDetectionResult** is a retroactive annotation step. It takes an already-loaded Protein via a ProteinConformation reference, reads PDB atom name strings at the loading boundary to determine which hydrogens are present on titratable sidechains, and sets `protonation_variant_index` on each Residue via const_cast through the non-const `MutableResidueAt()` accessor. This is the only code that writes protonation_variant_index after construction in the PDB path. In the ORCA path, the loader already sets it during construction, making the detection step a redundant validation. The const_cast is notable: it acknowledges that protonation_variant_index logically belongs on the Protein (it is identity, not geometry) but that the code was structured so that only a conformation-level result had the opportunity to set it.

**The Protonator interface** (PROPKA, KaML) takes a loaded Protein, a conformation for geometry, and a pH. It returns a ProtonationState, which is a value type holding per-residue decisions (variant_index, pKa, is_charged). The Protonator does not modify the Protein. It does not add hydrogen atoms. It does not assign charges. It decides. The PROPKA implementation writes a temporary heavy-atom PDB (writing to the string boundary), calls the propka3 binary, parses the `.pka` output (reading from the string boundary), and applies Henderson-Hasselbalch to map each pKa to a variant_index. The KaML implementation follows the same pattern. Both produce ProtonationState objects that sit as free-floating values with no connection back to the Protein that produced them.

**NamingRegistry** is the translation layer. It maps residue names between PDB canonical (HIS), AMBER (HID/HIE/HIP), and CHARMM (HSD/HSE/HSP) naming universes. It also translates atom names (H vs HN for the backbone amide, HB2/HB3 vs HB1/HB2 for beta methylenes). The OrcaRunLoader uses `ToCanonical()` to map AMBER prmtop residue labels back to canonical amino acid types. The PropkaProtonator uses `ThreeLetterCodeForAminoAcid()` to write PDB-format names when generating input. What the NamingRegistry does NOT yet do, but would need to, is produce a complete set of translated residue names for an entire protein to hand to tleap as input. The ResolveForTool method can do this one residue at a time given a canonical name, a ToolContext, and a variant key (e.g., "delta", "epsilon", "protonated"). The variant key comes from ProtonationVariant::registry_key on the AminoAcidType table. The data path is: ProtonationState.variant_index -> AminoAcidType.variants[idx].registry_key -> NamingRegistry.ResolveForTool(canonical, Amber, key) -> AMBER name. This chain is complete in the code. Nobody calls it end-to-end.


## 3. The Builder Gap

Neither loader does what the thesis requires: take an existing protein, decide on a different protonation, and produce a new Protein with the right hydrogen atoms for that protonation. The missing piece is the build step between "I have protonation decisions" and "I have a Protein with hydrogen atoms." That step is what tleap does externally for the ORCA pipeline, and what nobody does for the PDB pipeline.

The builder would need to perform these operations in sequence. First, extract heavy-atom positions and the residue sequence from the source, whether that is a PDB file, a prmtop+XYZ pair, or an existing Protein object. Second, if protonation is requested, run a Protonator (PROPKA or KaML) to get a ProtonationState, or accept one directly. Third, translate the ProtonationState into tool-specific residue names using NamingRegistry, write a temporary PDB with those names, and invoke the force field tool (tleap for AMBER, pdb2gmx for GROMACS) to produce a topology with hydrogen atoms at correct positions. Fourth, read back the tool's output (a prmtop and inpcrd, or a processed PDB) to get the protonated atom list and hydrogen positions. Fifth, build a new Protein from this data in the same way LoadOrcaRun builds from a prmtop today: create atoms, residues with variant_index set from the protonation decisions, call FinalizeConstruction to detect backbone indices, rings, and bonds. Sixth, create a CrystalConformation with the full positions (heavy atom positions from the source, hydrogen positions from the force field tool). Seventh, set ProteinBuildContext recording exactly what happened.

The critical data flow across the tool boundary is: NamingRegistry translates our canonical residue names plus variant keys into AMBER residue names. tleap receives a PDB with those names and a leaprc specifying ff14SB. tleap produces a prmtop (atom list with charges) and an inpcrd (coordinates). We read back the prmtop exactly as LoadWithPrmtop does today. The atom names, residue labels, and element ordering in the prmtop become our atom list. The hydrogen positions from inpcrd become part of our positions vector. Heavy atom positions come from the source, not from tleap, because tleap may minimise or adjust them, and we want to preserve the original geometry.

What NamingRegistry currently lacks for this is not data but orchestration. It can translate one residue name at a time. The builder would need to iterate all residues, look up each titratable residue's decision in the ProtonationState, find the registry_key from the variant, call ResolveForTool, and produce a complete PDB with the right residue names. This is straightforward code, not a new abstraction.


## 4. The Real Constraints

**Atom indices are load-bearing everywhere.** Every ring stores vertex atom indices. Every bond stores two atom indices. Every residue stores a vector of atom indices and backbone index slots (N, CA, C, O, H, HA, CB). ConformationAtom vectors are parallel to the atom list. SpatialIndexResult's neighbour lists use atom indices. Every calculator iterates by atom index. RingNeighbourhood and BondNeighbourhood store source indices. The index contract is so pervasive that changing the atom list means rebuilding the entire Protein from scratch, not patching it. This is consistent with the design intent: different protonation produces a new Protein, not a mutated one.

**Ring detection depends on protonation_variant_index.** In DetectAromaticRings, HIS residues check protonation_variant_index to decide between HidImidazoleRing, HieImidazoleRing, and HisImidazoleRing. If variant_index is not set before FinalizeConstruction is called, the code falls back to string-based hydrogen detection. This means the builder MUST set variant_index on each Residue before calling FinalizeConstruction. The OrcaRunLoader does this correctly. The PdbFileReader does not, because it has no protonation information at that point.

**FinalizeConstruction order is mandatory.** CacheResidueBackboneIndices must run first (rings need backbone cache to know residue types). DetectAromaticRings must run second (bonds need the aromatic atom set). DetectCovalentBonds must run third. Any builder must call FinalizeConstruction after all atoms and residues are added with their variant indices set.

**The Pipeline and all calculators are downstream.** RunClassicalCalculators takes a ProteinConformation and attaches results. It does not know or care how the Protein was built. The 8 calculators read from the Protein's typed objects (rings, bonds, atoms) and the conformation's positions. As long as the Protein is correctly constructed with correct rings, bonds, backbone indices, and atom list, the calculators produce correct results. The builder's output is the calculators' input. Nothing in the calculator layer needs to change.

**The variant index contract is validated at startup.** ValidateVariantIndices() checks that HIS variants[0] is HID, variants[1] is HIE, variants[2] is HIP, and similarly for all titratable amino acids. This runs in the test suite's SetUpTestSuite. Any code that uses variant_index must agree with this ordering, and it does: PropkaProtonator hardcodes the same mapping, ProtonationDetectionResult hardcodes it, and OrcaRunLoader reads it from the prmtop labels.


## 5. The Real Freedoms

**The loaders are not sacred.** LoadProtein and LoadOrcaRun are free functions returning result structs. They are called from tests and from batch processing code. Nothing in the core object model depends on their existence. They could become convenience wrappers around a builder without changing any downstream code. The key observation is that both loaders do the same final steps: create a Protein, add atoms and residues, call FinalizeConstruction, set ProteinBuildContext, create a CrystalConformation. These steps are the builder's Build() method.

**ProteinBuildContext is a plain struct with public fields.** It records provenance strings and is trivially constructible. The builder would set it with more complete information (protonation tool, pH, force field name, prmtop path) than either loader currently provides. LoadProtein sets only pdb_source. LoadOrcaRun sets pdb_source, force_field, protonation_tool, prmtop_path, and tleap_script_path. The builder would always set all relevant fields.

**Protein is default-constructible.** The loaders create empty Proteins and populate them via AddAtom and AddResidue. The builder would do the same. No change to the Protein class is needed for the builder to work. The builder does not need a copy constructor because it takes source data (heavy atom positions, sequence), not a Protein reference.

**ProtonationState is already a complete value type.** It holds per-residue decisions with variant_index, pKa, amino_acid, and is_charged. It has ForResidue() lookup, NetChargeForProtein(), Describe(), and equality comparison. It is the input the builder needs from the decision step and the output it can return alongside the Protein.

**The test fixtures create Proteins through the loaders.** PropkaTest, GmxChargeTest, KamlTest, and the Orca tests all call LoadProtein or LoadOrcaRun in their SetUp methods. If the builder wraps these loaders, the tests would call the builder instead. Since the builder produces the same output (a `unique_ptr<Protein>` with one conformation), the test assertions about atom counts, residue counts, variant indices, and charge totals would not change.


## 6. What the Builder Needs from tleap

tleap needs: a PDB file with correct AMBER residue names for each titratable residue, and a leaprc script specifying the force field (source leaprc.protein.ff14SB). It produces: a prmtop (AMBER topology with atom names, residue labels, charges, radii, and bonding information) and an inpcrd (coordinates of all atoms including placed hydrogens).

The NamingRegistry provides the residue name translation. For each residue in the protein, the builder would: (1) check if the ProtonationState has a decision for this residue, (2) if yes, look up the variant's registry_key from AminoAcidType.variants[decision.variant_index], (3) call NamingRegistry.ResolveForTool(canonical_name, ToolContext::Amber, registry_key) to get the AMBER name, (4) write that name into the PDB. For non-titratable residues, the canonical name passes through unchanged.

The builder then calls tleap as an external process, much as PropkaProtonator calls propka3. It writes a tleap input script (load PDB, save prmtop and inpcrd), runs it, reads the outputs. The prmtop reading code already exists in OrcaRunLoader's ReadPrmtopStrings and ReadPrmtopInts. The builder would reuse this directly. The inpcrd reading code does not exist but is trivial: AMBER inpcrd is a fixed-format text file with 6 coordinates per line.

A subtlety: tleap places hydrogens based on its internal geometry rules, which may not match the source PDB's hydrogen positions exactly. The builder should take heavy atom positions from the source and hydrogen positions from tleap's inpcrd. This requires matching atoms between the source and the tleap output, which the NamingRegistry's atom name translation facilitates. However, tleap also reorders atoms (hydrogens are interleaved with their parent heavy atoms in tleap output), so the builder must build its atom list from the prmtop ordering, not the source ordering. This is exactly what LoadWithPrmtop already does.


## 7. Test Impact

The 274 tests (now described as 266 in some documents) create Proteins through three patterns.

**Pattern 1: LoadProtein from PDB files.** The PropkaTest fixture, the GmxChargeTest fixture, the KamlTest fixture, and various calculator tests load from PDB paths. These produce Proteins with whatever atoms the PDB has, no protonation step. If the builder wraps LoadProtein as "builder with PDB source and as-is protonation," these tests would call the builder instead of LoadProtein directly. The resulting Protein would be identical because the builder would internally do exactly what LoadProtein does. No test assertions would change.

**Pattern 2: LoadOrcaRun from prmtop+XYZ.** The OrcaRunTest tests and the 723-protein batch validation use this path. Again, if the builder wraps LoadOrcaRun as "builder with prmtop+XYZ source and as-is protonation," the output is identical. The batch runner iterates directories and calls LoadOrcaRun; it would call the builder instead.

**Pattern 3: Direct Protein construction in unit tests.** Some calculator tests (not in the protonation test file, but implied by the spec's 274 tests) may create small Proteins directly for analytical geometry tests. These would not go through the builder because they are constructing synthetic test geometries. The builder need not be the ONLY way to create a Protein for testing purposes; it is the single entry point for production protein loading. Test code that creates a three-atom Protein for verifying a Biot-Savart analytical formula would continue to use AddAtom and AddResidue directly.

The critical question is whether any test depends on the internal mechanics of how LoadProtein or LoadOrcaRun work, rather than on what they produce. From reading the test file, the answer is no. The tests check output properties: atom counts, residue counts, charge totals, variant indices, shielding tensor consistency. They do not check intermediate states of the loading process. This means the builder can replace the loaders without breaking any existing test, provided it produces the same Protein for the same inputs.

The one area of concern is ProtonationDetectionResult's const_cast through `const_cast<Protein&>(conf.ProteinRef())`. This writes protonation_variant_index after the Protein is "complete." Under the builder, variant_index would be set during construction, making the detection step purely a validation pass. The test `ProtonationDetectedFromStructure` verifies that detection works on an already-annotated protein, which would still pass because detection reads hydrogen atoms regardless of whether variant_index is already set. The test `ProtonationDetectionOnCharmmNaming` does the same for CHARMM-named structures. Both would pass unchanged.


## Contradictions Between Constitution and Code

The constitution says "one protonation state per ProteinConformation." The code puts protonation_variant_index on Residue, which is owned by Protein, not by ProteinConformation. ProteinConformation.h declares a `protonation` field of type ProtonationState in the constitution's object model table, but the actual header has no such field. The code is correct: protonation determines the atom list, which is per-Protein. The constitution's text here is stale.

The constitution describes a copy-and-modify pattern with a proper copy constructor. The code has `Protein(const Protein&) = delete`. There is no Copy() factory method. The constitution's Copy() mechanism was never built. The builder spec correctly identifies this as something that does not need to exist: the builder takes source data and constructs a new Protein, rather than copying and mutating an existing one.

The constitution describes typed conformation factory methods including AddNMRConformation, AddMDFrame, AddPrediction, AddMinimised. The code has only AddCrystalConformation and AddDerived. The other factory methods were not needed yet. This is an incomplete implementation, not a contradiction; the base ProteinConformation is fully functional and handles all current use cases.

The constitution says "no agent creates a ProteinConformation directly." Both loaders call `protein->AddCrystalConformation()`, which is the factory method on Protein, so this rule is followed. The ProteinConformation constructor is public (it must be, for make_unique in the factory methods), but nothing outside of Protein's factory methods calls it directly in production code.

The OBJECT_MODEL.md says Protein is "non-movable and non-copyable" and "lives on the heap via unique_ptr, created by LoadProtein() which returns unique_ptr<Protein> in a LoadResult." This matches the code exactly. The Protein class deletes copy, move, copy-assign, and move-assign. Both LoadProtein and LoadOrcaRun return result structs containing `unique_ptr<Protein>`.
