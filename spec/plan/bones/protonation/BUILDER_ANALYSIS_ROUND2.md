# Builder Gap Analysis: Supplementary (Round 2)

This document addresses five specific gaps identified in the critique of the round 1 analysis. It does not repeat what round 1 got right. It digs into the places round 1 gestured at but did not trace through.


## 1. Heavy atom ordering through tleap and back

The round 1 analysis correctly noted that LoadWithPrmtop builds its atom list from prmtop ordering, and that NamingRegistry "facilitates" matching. It did not trace through what actually happens when you read a prmtop.

LoadWithPrmtop in OrcaRunLoader.cpp reads four prmtop sections: ATOM_NAME (4-char fields), RESIDUE_LABEL (4-char fields), RESIDUE_POINTER (1-based indices into the atom list), and ATOMIC_NUMBER. The RESIDUE_POINTER array defines where each residue's atoms begin. The atoms between RESIDUE_POINTER[r] and RESIDUE_POINTER[r+1] belong to residue r. Within each residue, tleap outputs atoms in a fixed order determined by the force field library: the heavy atom backbone (N, CA, C, O) first, then sidechain heavy atoms in library order, with hydrogen atoms interleaved immediately after the heavy atom they are bonded to. So for alanine you get N, H, CA, HA, CB, HB1, HB2, HB3, C, O. The heavy atoms are not contiguous.

The current code handles this correctly because it takes positions directly from the XYZ file that tleap also produced. The XYZ and prmtop are paired outputs of the same tleap run, so atom i in the XYZ is atom i in the prmtop. The code iterates all n_atoms from the prmtop, creates Atom objects with the prmtop's atom names and elements, assigns residue membership from the RESIDUE_POINTER ranges, and takes positions from the corresponding XYZ entry. No matching is needed because the data sources are already aligned.

The builder changes this picture fundamentally. When the builder writes a PDB with protonation-specific residue names (HID instead of HIS, ASH instead of ASP) and feeds it to tleap, it gets back a prmtop and inpcrd. The inpcrd contains all positions in prmtop order, but the heavy atom positions came from tleap's interpretation of the input PDB, not from our source protein. The critical question is whether changing a residue name changes the heavy atom ordering within that residue.

For the standard residues, the answer is that it depends on the library file. In ff14SB's amino acid libraries, HID, HIE, and HIP are independent residue templates. They share the same heavy atom backbone (N, CA, C, O, CB, CG, ND1, CD2, CE1, NE2) but differ in which hydrogens are present and where they appear in the interleaved order. The heavy atom names and their relative ordering within the residue are the same across HIS tautomers. Similarly, ASP and ASH share the same heavy atoms (the proton on ASH is an additional hydrogen, not a replacement of a heavy atom). GLU and GLH, LYS and LYN, CYS and CYM all follow the same pattern: the heavy atom skeleton is invariant across protonation variants, and only hydrogen count changes.

This means that for the builder's coordinate substitution problem, heavy atoms can be identified by name within each residue. The prmtop tells you the atom name and the ATOMIC_NUMBER for every atom. Any atom with ATOMIC_NUMBER != 1 is heavy. For each residue, the builder can construct a map from heavy atom name to source position, then walk the prmtop atoms and substitute: heavy atoms get source positions, hydrogens get inpcrd positions. The atom name matching is unambiguous within a residue because PDB atom names are unique per residue by convention, and ff14SB's libraries follow this convention strictly.

The one case that deserves scrutiny is CYX (disulfide). When CYS becomes CYX, the SG atom loses its HG hydrogen but the heavy atoms remain identical. However, tleap may adjust the SG position during disulfide bond formation if it runs a minimisation step. The builder should use the source SG position regardless, since the source geometry is authoritative and tleap's geometry adjustments are unwanted.


## 2. tleap failure modes and force field coverage

The round 1 analysis did not discuss what happens when tleap rejects the protonation state the builder hands it. This matters because the variant table in AminoAcidType includes ARN (deprotonated arginine), which PROPKA can predict at extremely high pH or in unusual electrostatic environments.

The ff14SB leaprc does not include a library for ARN. The standard AMBER protein force fields (ff14SB, ff19SB, ff03) parametrise ARG as always protonated because its pKa of approximately 12.5 means it is essentially always cationic at biological pH. If the builder writes a PDB with residue name ARN and feeds it to tleap with source leaprc.protein.ff14SB, tleap will fail with an unknown residue type error. The same problem affects TYM (deprotonated tyrosinate). While tyrosinate parameters do exist in some AMBER parameter sets, they are not in the standard ff14SB leaprc.

CYM (deprotonated cysteine thiolate) outside a disulfide context is a more interesting case. AMBER does have CYM parameters in ff14SB for the specific case of a metal-coordinated or free thiolate cysteine. But CYM requires careful handling: the charge on the deprotonated thiolate is -1, and tleap needs to know this is not a disulfide. If PROPKA predicts a free CYS should be deprotonated (pKa around 8, so at pH above 9 this is plausible), the builder can write CYM and tleap will accept it. But if the CYS is actually in a disulfide (which PROPKA cannot detect from pKa alone), writing CYM instead of CYX will produce wrong topology.

This reveals a dependency the builder must handle: disulfide detection must happen before protonation variant assignment. The builder needs to identify disulfide-bonded cysteines from the source structure's geometry (SG-SG distance below approximately 2.5 angstroms), mark them as CYX regardless of what the Protonator says, and only apply the Protonator's CYS decision to free cysteines. ProtonationDetectionResult already has this logic (it checks disulfide bonds), but it runs after construction. The builder needs the same check before writing the tleap input PDB.

The practical consequence is that the builder must have an error path for tleap failure. The PROPKA or KaML decision might produce a variant that the force field cannot handle. The builder should detect this before calling tleap (by checking that the NamingRegistry's ResolveForTool produces a name that ff14SB recognises) and report which residue and variant caused the problem. Silently falling back to a default protonation would defeat the purpose of protonation-specific analysis.


## 3. ProtonationDetectionResult after the builder exists

The round 1 analysis noted that under the builder, variant_index would be set during construction, making detection "purely a validation pass." This is true for the builder path but misses the genuine use case where detection is the primary source of truth.

When loading a pre-protonated structure that was not built by our system (the AsIs path in the builder spec), the builder has no ProtonationState to draw from. It receives a PDB or prmtop that some external tool protonated, and the builder needs to figure out what protonation is present. This is exactly what ProtonationDetectionResult does: it reads hydrogen atom names to determine which titratable variant each residue has.

In the ORCA/prmtop path today, the loader reads AMBER residue labels (HID, HIE, HIP) directly from the prmtop, so it knows the variant from the topology file rather than from hydrogen atom inspection. ProtonationDetectionResult is then run as validation and always agrees (the test ProtonationDetectedFromStructure confirms this). But if someone loads a PDB that was protonated by pdb2gmx, Reduce, or any other tool that does not annotate residue names with variant information, detection from hydrogen names is the only way to determine what was done.

The more subtle question is about timing. FinalizeConstruction uses protonation_variant_index for HIS ring detection. If the builder sets variant_index before FinalizeConstruction (as LoadWithPrmtop does), ring types are correct from the start. If instead the builder relies on ProtonationDetectionResult to set variant_index, that detection must run before FinalizeConstruction, which means before bonds are detected (since FinalizeConstruction includes bond detection). But ProtonationDetectionResult as currently written uses BondCategory::Disulfide to identify CYX, and bonds are detected by FinalizeConstruction. This is a circular dependency: detection needs bonds for CYX, but ring detection within FinalizeConstruction needs detection results for HIS.

For the builder, this circularity does not arise if it sets variant_index from the ProtonationState (builder path) or from the prmtop labels (ORCA path) before calling FinalizeConstruction. But for the AsIs-with-unknown-source path, the builder would need to break the sequence: run CacheResidueBackboneIndices and DetectCovalentBonds first (for disulfide detection), then run protonation detection (which needs bonds), then set variant_index, then run DetectAromaticRings. This means FinalizeConstruction's current fixed order (backbone, rings, bonds) would need to become two phases for this specific path.


## 4. The AminoAcidType atom list and its single-variant nature

The HIS entry in AminoAcidType.cpp lists atoms including HE2 (the epsilon nitrogen hydrogen) but not HD1 (the delta nitrogen hydrogen). This is the HIE atom set. The round 1 analysis did not flag this. The critique asks whether any code path uses this atom list to decide what atoms should exist.

Tracing every use of AminoAcidType::atoms in the codebase reveals exactly two consumers. The first is IupacAtomIdentity.cpp, which builds a lookup table from the atoms list at static initialisation time. The table maps (residue_code, atom_name) to IupacAtom structs that carry element, is_backbone, and is_ring_member flags. This is used by EnrichmentResult to classify atoms (backbone, sidechain, aromatic) without string comparisons at calculator time. The second consumer is Protein::CacheResidueBackboneIndices, which uses chi_angles (not atoms directly) to populate chi angle indices, and that code iterates the protein's actual atoms, not the AminoAcidType template.

Critically, neither consumer uses the atoms list to determine what atoms a residue should have. IupacAtomIdentity uses it for classification: given an atom that already exists on the protein, look up its role. If a HID residue has HD1 but not HE2, the lookup for HD1 returns nullptr (since HD1 is not in the HIE atom list), and the atom gets classified with default properties. This is a minor gap in enrichment, not a construction error.

The ring detection code does not use the atoms list at all. It uses AminoAcidType::rings, which for HIS lists the five ring atom names (CG, ND1, CE1, NE2, CD2). These are all heavy atoms present in every HIS tautomer. The ring definition is protonation-independent. Whether the ring becomes HidImidazoleRing, HieImidazoleRing, or HisImidazoleRing depends solely on protonation_variant_index, not on the atoms list.

So the single-variant atom list is a cosmetic incompleteness, not a functional bug. For the builder, it means IupacAtomIdentity may fail to classify HD1 on a HID residue, giving it default enrichment properties. This could be fixed by adding per-variant atom lists to AminoAcidType, but the impact is limited to enrichment classification of one hydrogen atom per HID histidine, and the calculators that consume enrichment data (ring current calculators check ring membership from Ring::atom_indices, not from IupacAtomIdentity) are unaffected.


## 5. The coordinate matching problem in concrete detail

The round 1 analysis said "take heavy atom positions from the source and hydrogen positions from tleap's inpcrd" without tracing how this identification works mechanically. This is the hardest practical problem the builder faces.

The prmtop provides ATOMIC_NUMBER for every atom. Hydrogen has atomic number 1. Everything else is heavy. So for any atom in the prmtop, the builder can immediately classify it as heavy or hydrogen. The inpcrd provides positions in the same order as the prmtop, so reading (ATOMIC_NUMBER, inpcrd position) pairs gives a classified position for every atom.

For hydrogens, the builder takes the inpcrd position directly. tleap placed them, and they have no source equivalent.

For heavy atoms, the builder needs to substitute the source protein's position. The source protein's atoms are in source order (PDB file order or existing Protein atom vector order), and the prmtop's atoms are in tleap library order. These orderings are different. Within a residue, the source PDB might list atoms as N, CA, C, O, CB, CG, ND1, CD2, CE1, NE2 (the PDB convention), while tleap lists them as N, CA, C, O, CB, CG, ND1, CD2, CE1, NE2 (the same names, but potentially in a different sequence depending on the library). In practice, ff14SB's library ordering for backbone atoms is N, CA, C, O, which matches PDB convention, and sidechain atoms follow the library's internal tree traversal order, which usually but not always matches PDB order.

The reliable matching strategy is by atom name within each residue. The builder knows the residue correspondence: residue r in the source maps to residue r in the prmtop (tleap preserves residue sequence order from its input PDB). For each residue, the builder builds two maps: source_name_to_position (from the source protein's atoms in that residue) and prmtop_name_to_index (from the prmtop's ATOM_NAME entries in that residue's range). For each heavy atom in the prmtop residue, it looks up the atom name in the source map. If found, it substitutes the source position. If not found, something has gone wrong (a heavy atom in the prmtop that does not exist in the source), and the builder should fail with a diagnostic.

The atom names between source and prmtop should agree because the builder wrote the tleap input PDB using the source's atom names, and tleap preserves heavy atom names from the input (it renames them only if they conflict with the library, which does not happen for standard amino acids with standard PDB naming). If the source used non-standard atom names (CHARMM naming via the fleet data, for instance), the builder should translate them through NamingRegistry before writing the tleap input PDB, ensuring that tleap receives and outputs standard AMBER atom names.

There is one subtlety about tleap's geometry modification. By default, tleap does not minimise or adjust heavy atom positions when loading from a PDB. It takes heavy atom coordinates directly from the input and only generates hydrogen coordinates using its internal geometry rules. So the inpcrd's heavy atom positions should be identical to what the builder wrote into the tleap input PDB, which came from the source. In that case, the coordinate substitution is technically unnecessary for heavy atoms because they already match. However, the builder should still perform the substitution explicitly rather than trusting this invariant, because tleap's behaviour might change across versions, and the principle of taking authoritative data from its authoritative source (heavy atoms from the original structure, hydrogens from the force field tool) is worth encoding in the code regardless of whether it happens to be a no-op today.

The positions vector for the new Protein is then assembled in prmtop order: for each atom in the prmtop, the position is either the source heavy atom position (matched by name within the residue) or the tleap hydrogen position (from the inpcrd). This vector, together with the prmtop's atom names, elements, and residue structure, is exactly what LoadWithPrmtop already consumes. The builder's output stage is LoadWithPrmtop called with the builder's own prmtop and a position vector it assembled from two sources rather than a single XYZ file.
