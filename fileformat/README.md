# Analysis H5 File Format — Standalone Serialisation Library

Standalone C++ object model for the analysis H5 files produced by
`nmr_extract --trajectory --analysis`. Created 2026-04-15 as a
prototype outside the main project, validated against the full
10-protein fleet calibration dataset.

## Build

Requires: C++17, HDF5 C library (system), HighFive headers (from
`../extern/HighFive/`).

```
cd fileformat
mkdir -p build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)
```

Produces:
- `libanalysis_file.a` — the serialisation library
- `roundtrip_test` — 6-phase validation exe
- `roundtrip_write` — generates persistent roundtripped H5 files

## Test

### Prerequisites

Copy the 10 fleet H5 files into `fileformat/test/`:

```
for p in 1B1V_4292 1BWX_3449 1CBH_192 1DV0_4757 1G26_4656 \
         1HA9_5028 1HD6_4820 1HS5_4934 1I2V_4976 1I8X_4351; do
  cp /shared/2026Thesis/fleet_calibration-working/${p}/analysis_output/${p}_analysis.h5 \
     fileformat/test/
done
```

### Phase 1: C++ roundtrip (all 6 validation phases)

```
cd build
./roundtrip_test ../test
```

Tests per file:
1. **Dataset byte comparison** — read H5, write back, compare every
   dataset byte-for-byte using `read(ptr, dtype)` (true raw bytes,
   not the type-converting `read(char*)` overload).
2. **Double roundtrip** — read the written file, write again, compare
   the two written files. Proves writer determinism.
3. **Object model identity** — compare every field of the two
   in-memory `AnalysisFile` objects via `memcmp`/`==`.
4. **Structural coverage** — bidirectional check that no datasets,
   groups, or attributes are missing or extra in either direction.
5. **1CBH aim dtype exception** — verifies float64 aim is normalised
   to float32, conversion is lossless, shape preserved.
6. **Special value census** — counts NaN, Inf, -Inf, negative zero,
   and subnormals in every float dataset; verifies counts match.

Expected: 10/10 pass. 186 datasets, 193 object fields, 68 structural
checks, 136 special-value checks per file.

### Phase 2: NPY cross-validation

```
cd build
./roundtrip_write ../test ../test/rt
cd ..
python3 npy_cross_validate.py /shared/2026Thesis/fleet_calibration-working test/rt
```

Compares the roundtripped H5 files against the NPY snapshot files
written independently by the C++ calculators at ~1ns intervals.
This validates the entire chain:

```
Calculator -> NPY                               (direct)
Calculator -> AnalysisWriter -> H5 -> fileformat -> H5  (roundtrip)
```

10 proteins x 26 snapshots x ~111 checks = 29,250 comparisons.
Expected: 29,250 pass, 0 fail, 520 skip.

## H5 Schema Summary

186 datasets across 21 groups. All files share identical structure;
dimensions vary per protein.

| Dimension | Range across fleet |
|-----------|-------------------|
| T (frames) | 626 (all) |
| N (atoms) | 335 -- 712 |
| R (residues) | 23 -- 45 |
| B (bonds) | 339 -- 653 |
| n_rings | 3 -- 7 |

Dtypes: float64 (most), float32 (aim embedding only), int32, int16,
int8, variable-length strings. SphericalTensor datasets (24 of them)
carry `layout` and `convention` attributes.

## Validated Findings

### Fileformat library

The library roundtrips all 186 datasets byte-identically, including:
- NaN, Inf, -Inf, negative zero, subnormal values
- Variable-length string datasets (atom names, residue names, etc.)
- All group and dataset attributes
- int8/int16/int32/float32/float64 dtypes preserved exactly

The 1CBH_192 float64 aim exception is handled: normalised to float32
on read, proven lossless across all 79,486,976 values (the original
data was float32 that had been upcast to float64 before writing).

### Main program findings (from NPY cross-validation)

These are properties of the main project's AnalysisWriter, not bugs
in the fileformat library. Documented here for the session that
investigates them.

**Ring geometry normals differ between NPY and H5 paths.** The NPY
`ring_geometry.npy` is written by `ConformationResult::WriteAllFeatures`
using `Ring::ComputeGeometry` (cross-product of two ring diagonals).
The H5 `ring_geometry/data` is written by `AnalysisWriter` using
`GeometryResult` (which computes ring geometry independently).
Center coordinates and radii are byte-identical; normals diverge by
up to ~18 degrees on thermally distorted rings (dot product as low as
0.95). This is not a data corruption — it is two different algorithms
computing the normal of a non-planar ring.

**Chi dihedral cos/sin differ by 1 ULP.** The NPY `dssp_chi.npy` is
computed by `DsspResult` (which gets chi angles from the DSSP library).
The H5 `dihedrals/chi*_cos` and `chi*_sin` are computed by
`AnalysisWriter::Dihedral()` from raw atom positions. Two independent
trig evaluations of the same angle give results within 1 ULP
(~1e-16), which is machine epsilon.

**Four GROMACS energy terms not harvested into H5.** The NPY
`gromacs_energy.npy` has 43 columns. The H5 `energy/` group stores
39 of them. Missing: `coulomb_14` (col 2), `lj_14` (col 10),
`disper_corr` (col 11), `total_energy` (col 14).

**`aromatic_E_magnitude` not harvested.** `coulomb_scalars.npy` col 3
(`ca.aromatic_E_magnitude`) is written to NPY but not stored in H5.

**Ion distance sentinel differs.** NPY `hydration_shell.npy` col 2
stores `+inf` when no ion is within cutoff. H5 `water/nearest_ion_dist`
stores `-1.0`. The AnalysisWriter transforms the sentinel at harvest
time (`AnalysisWriter.cpp:217-218`).

**DSSP per-residue vs per-atom.** NPY DSSP arrays are per-atom
(residue values broadcast to all atoms in the residue). H5 stores
per-residue (T, R). The cross-validation broadcasts H5 per-residue
to per-atom and confirms identity.

**Coulomb NPY files absent.** The fleet runs use `--no-coulomb`
(APBS replaces Coulomb). No `coulomb_shielding.npy` etc. in the
snapshot directories. The H5 `efg/coulomb_*` datasets are present
(populated from CoulombResult, which runs even with APBS) but cannot
be cross-validated against NPY.

### Critical test methodology note

The HighFive `read(T*)` overload deduces the memory type from `T`.
When `T = char` (as in `vector<char>`), HDF5 converts each element
to a single byte. For float64 data, this compares only 1/8 of the
actual bytes and silently passes on corrupted data. The roundtrip
test uses `read(ptr, dtype)` which specifies the file dtype as the
memory type, forcing a true raw-bytes copy. Any future H5 comparison
code must use this two-argument form.
